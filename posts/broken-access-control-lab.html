  • Initial Vulnerable Build: broken-access-api-lab-v1

    • Broken Access Control Lab

    This project was built as a vulnerable Node.js REST application using Express, sessions, EJS views and JSON data storage. The goal was to intentionally create common access control failures, exploit them, then harden the application using proper authorization checks.

    The application starts with a simple login portal and multiple test users: alice, bob and admin.

    Initial Vulnerabilities

    After login, regular users could directly access links that should have required ownership checks or elevated privileges.

    The dashboard exposed direct object references for users, orders and admin routes.

    BOLA / IDOR User Enumeration

    Alice was able to browse directly to another user record by changing the ID in the URL. No ownership validation was performed server-side.

    Sensitive data such as usernames, email addresses and roles were disclosed. This is a classic Broken Object Level Authorization issue.

    Broken Order Ownership Controls

    The same flaw existed on order objects. Alice could request Bob's order simply by changing the order ID.

    The response leaked ownership data, purchased item names and pricing information.

    Vertical Privilege Escalation

    Regular users could access administrative reports directly without having an admin role.

    This exposed internal business metrics such as revenue totals and user counts.

    Mass Assignment via Burp Suite

    Using Burp Repeater, a normal user modified a profile update request and injected an unauthorized role field.

    Because the API trusted all submitted fields, Alice's role changed from user to admin.

    This demonstrates privilege escalation through mass assignment.

    Security Hardening

    The application was then rebuilt with secure authorization controls:

    - Ownership checks for user records
    - Ownership checks for orders
    - Admin-only middleware for privileged routes
    - Allowlisted profile fields only
    - 403 responses for unauthorized requests

    Alice can still view her own profile, but no longer other users.

    Verification Testing

    All previously vulnerable endpoints were re-tested after remediation.

    Attempting to access Bob's user profile now returns: {"error":"Access denied"}

    Attempting to access Bob's order now returns: {"error":"Access denied"}

    Attempting to open admin reports as a normal user now fails.

    Burp Suite mass assignment retesting also fails. Supplying: {"role":"admin"} no longer changes the user role.

    The response shows role remains: user

  • Final Hardened Build: broken-access-api-lab

    • Conclusion

    This lab demonstrated how Broken Access Control appears in modern web APIs through BOLA / IDOR, privilege escalation and mass assignment flaws. The vulnerable version allowed unauthorized access to users, orders and admin data. The hardened version implemented proper authorization middleware, ownership checks and secure field allowlisting to eliminate those attack paths.
