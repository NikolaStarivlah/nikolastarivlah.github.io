Vulnerability 1: Broken Access Control

The /admin endpoint is accessible without enforcing authentication or authorization checks.

The admin panel was accessed directly at:

http://localhost:3000/admin

The page loaded successfully without authentication, displaying:

- Current user: Not logged in
- Current role: guest

This confirms that access control is not enforced at the application level.

Attack

An attacker can directly access /admin without logging in.

Even after logging in as a normal user (user / user123), the admin page remains accessible.

Even though default values such as null (user) and "guest" (role) are assigned, no validation logic exists to restrict access.

This allows exposure of sensitive content such as “Top Secret Data”.

Fix

if (!req.session.user || req.session.role !== "admin") {
    return res.status(403).send("Access denied");
}

The fix enforces both authentication and role-based access control.

Unauthorized users now receive a 403 response.

Only admin users can access the admin panel.

Vulnerability 2: Reflected XSS

The /search endpoint reflects user input without proper encoding.

Attack

http://localhost:3000/search?q=<script>alert(1)</script>

More advanced payload:

Nginx Reverse Proxy

The application was transferred to Ubuntu using SCP and executed with Node.js.

Nginx was configured as a reverse proxy.


server {
    listen 80;

    location / {
        proxy_pass http://localhost:3000;
    }
}

The application is now accessible externally.

Nginx Detection & Blocking

sudo tail -f /var/log/nginx/access.log

Initial rule:


if ($query_string ~* "script") {
    return 403;
}

This failed due to URL encoding:


%3Cscript%3Ealert(1)%3C/script%3E

Updated rule:


if ($request_uri ~* "%3Cscript%3E") {
    return 403;
}

Nginx blocks the request with 403.

The request is still logged for analysis.

Conclusion

This shows a layered defense approach combining detection and prevention.
