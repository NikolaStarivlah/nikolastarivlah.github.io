  • Initial Vulnerable Build: auth-session-v1

    • Auth Session Lab

    This project was built as a simple Node.js authentication lab using Express, express-session, EJS and body-parser. The goal was to intentionally create weak authentication and session logic, then improve it step by step.

    The screenshots above show the project structure, dependency installation, and the application running locally on http://localhost:3000.

    Username Enumeration

    The original login flow revealed valid usernames through different responses. A fake username returned a different message than a real username with the wrong password.

    When a non-existent account is submitted, the application returns User not found.

    When the valid username admin is submitted with an incorrect password, the application returns Incorrect password. Because these messages are different, an attacker can distinguish valid accounts from invalid ones and build a more efficient brute-force target list.

    Session Weakness

    After a successful login, the browser receives a session cookie. If a session token is stolen or reused, it can allow account takeover.

    The application used weak session handling with saveUninitialized: true and did not regenerate the session after login.

    Session Fixation

    Session fixation happens when an attacker sets a session first, the victim logs in, and the attacker reuses the same session ID.

    Because the session ID was trusted before authentication, the attacker could reuse the same cookie after the victim logged in.

    Access Control

    The admin route used weak authorization logic. This meant access checks depended only on session values.

    Weak access control combined with session issues can make role protection meaningless.

    Security Improvements

    The lab was then upgraded to demonstrate common defensive controls. Middleware was added for:

    - Rate limiting to slow brute-force attempts
    - Account lockout after repeated failures
    - Login logging for visibility
    - Generic login errors to stop username enumeration

    Instead of separate messages, the login now returns: Invalid credentials

    Authentication events were written into auth.log including successful logins, failures, rate-limit blocks and lockouts.

    Metrics Dashboard

    A metrics page was created to summarize attack activity and login behavior.

    The first version exposed metrics publicly, so access control was added to restrict the page to authenticated admin users only.

    The improved dashboard now shows:

    - Successful logins
    - Failed logins
    - Rate limit blocks
    - Lockouts triggered
    - Most targeted username

    Browser Logs Dashboard

    Instead of opening log files in VS Code, a browser route was added to read the log file directly through the application.

    This creates a lightweight monitoring panel for reviewing suspicious activity.

  • Final Hardened Build: auth-session-lab

    • Conclusion

    This lab demonstrated both offensive and defensive authentication concepts. It started with username enumeration, weak sessions and role abuse, then improved into a monitored authentication system with logging, rate limiting, lockouts and admin-only dashboards.
