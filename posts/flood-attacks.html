Overview

This writeup focuses on flood-based denial-of-service behavior from a defensive perspective. Rather than focusing on exploitation, the goal is to understand what the traffic looks like, how the target system behaves, and how a defender could detect abnormal packet patterns in a lab environment.

Writeup style: red headings mark major sections, blue headings break the analysis into smaller steps, and command/output blocks are used to document observations clearly.

ICMP Flood

Lab Setup

Two virtual machines were configured in an isolated host-only network. Kali Linux acted as the attacker machine and Ubuntu Server acted as the target system. 

Attacker: Kali Linux
Target: Ubuntu Server
Network: Host-only / isolated lab environment

What is the attack?

An ICMP flood is a denial-of-service technique that overwhelms a target with repeated ICMP Echo Requests, forcing the system to process and respond to excessive ping traffic.

How is it used by an attacker?

An attacker sends a large volume of ICMP packets toward a host to increase resource consumption, generate unnecessary replies, and create network congestion.

What would I see as a defender?

As a defender, I observed repeated ICMP Echo Requests and Echo Replies in packet captures. Traffic appeared at a sustained and abnormal rate from a single source IP. System monitoring also showed increased network activity and potential resource strain.

How would I detect it?

I would look for a sustained spike in ICMP traffic beyond normal baseline levels, especially when originating from a single source. In this lab, I used tcpdump to observe the packet flow and planned detection logic around repeated high-frequency ICMP activity. 

sudo tcpdump -i any icmp

System Impact Observed

Using top on the Ubuntu target, I monitored CPU usage and load during packet generation to identify whether the attack caused measurable system strain. 

top

The key things I watched for were increased CPU usage, rising load averages, and a sustained increase in packet-processing activity while the ICMP traffic was active.

Detection Engineering

I implemented a Snort IDS rule to detect ICMP flood behavior based on packet-rate thresholds and validated the rule using simulated attack traffic in the lab. 

alert icmp any any -> $HOME_NET any (
    msg:"ICMP Flood Detected";
    detection_filter:track by_src, count 100, seconds 1;
    sid:1000001;
    rev:1;
)

This approach is stronger than just saying “I sent a lot of pings.” It shows the lab setup, attacker objective, defender visibility, impact analysis, and how the behavior can be translated into detection logic.
